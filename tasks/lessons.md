# Lessons

- 사용자가 "클릭했는데 반응이 없다"고 말하면 입력 함수만 읽고 끝내지 말고, CSS 오버레이와 작은 히트 영역이 클릭을 먹는지 같이 확인한다.
- 레인/적 같은 실시간 전투 UI는 정확한 타깃 클릭만 요구하지 말고, 근처 큰 클릭 영역과 보조 액션 경로를 같이 제공한다.
- 실시간 화면을 `innerHTML`로 자주 다시 그리는 구조에서는 `click` 이벤트가 DOM 교체로 유실될 수 있으니, `pointerdown` 같은 즉시 입력이나 부분 갱신 구조를 우선 검토한다.
- 난이도 평가는 설정값 점수표만으로 끝내지 말고, 실제 규칙 위에서 `공격만`, `수리만`, `무반응`, `최적 추정` 같은 플레이 정책을 직접 돌려 퇴행 전략이 통하는지 확인한다.
- 실시간 이펙트를 추가한 뒤에는 상태 변경만 확인하지 말고, DOM 재생성과 애니메이션 수명을 함께 점검해 실제 화면에 보이는 텍스트/버스트가 남는지 검증한다.
- 한글 로컬라이징은 영어 키를 옮겼는지만 보지 말고, 실제 게임 화면에서 읽었을 때 자연스러운 한국어 UI/로그 문장인지 한 번 더 소리 내어 점검한다.
- 번역을 다듬을 때는 `워커`, `러너`처럼 플레이어가 이미 익숙한 장르 용어를 억지로 순화하지 말고, 어색한 건 설명 문장과 직역체 표현부터 손본다.
- 병렬로 여러 게임을 만드는 세션에서는 공용 `tasks/todo.md`를 새 작업에 재사용하지 말고, `tasks/todo-<game-slug>.md`를 만들어 각 게임의 계획과 리뷰를 분리한다.
- UI 패치를 끝내기 전에 반드시 데스크톱 기준 `한 화면 플레이`를 먼저 점검하고, 핵심 루프 영역이 페이지 세로 스크롤 없이 보이는지 확인한다.
- 카드형 UI는 정보량이 늘어날수록 개별 카드 설명을 계속 쌓지 말고, 먼저 `폭에 맞는 밀도`와 `읽는 순서`를 줄여서 선반이나 로그가 깨져 보이지 않게 만든다.
- 준비 페이즈처럼 플레이어가 아직 행동에 들어가지 않은 화면에서는 같은 안내 문장을 `콜아웃`, `중앙 박스`, `우측 패널`, `로그`에 반복하지 말고, 한 곳만 문장으로 두고 나머지는 숫자와 짧은 라벨로 축약한다.
- 좌측 인벤토리처럼 품목 수가 늘어나는 영역은 카드형을 고집하지 말고, 먼저 `행 리스트`가 맞는지 검토한다. 카드 크기만 줄이면 결국 세로 길이와 버튼 밀집 문제가 다시 나온다.
- 사용자가 `더 단순하게`, `한눈에`를 말하면 텍스트 수만 줄이지 말고 표현 형식 자체를 바꿔야 한다. 선반은 표/리스트, 로그는 카드가 아니라 콘솔형 스트림이 더 맞는지 먼저 본다.
- `hidden` 클래스로 화면 전환을 제어할 때는 ID 선택자나 별도 `display` 규칙이 이를 덮어쓰지 않는지 먼저 확인한다. 특히 `#screen-id` 규칙을 같이 쓰면 `display: none`이 먹지 않아 타이틀과 게임 화면이 동시에 보일 수 있다.
- 카드 배틀처럼 핵심 입력이 손패에 있는 게임은 그 입력 영역을 독립 하단 섹션으로 밀어두지 말고, 중앙 플레이 패널 안의 고정 하단 영역으로 붙여 한 화면 안에서 적 정보와 함께 보이게 해야 한다.
- Canvas 게임에서 `aspect-ratio`만으로 중앙 플레이 영역 크기를 잡지 말고, 사이드 패널과 헤더를 뺀 실제 가용 공간을 기준으로 프레임 크기를 맞춰야 한다. 폭 기준 비율만 쓰면 데스크톱에서 한 화면 비율이 쉽게 틀어진다.
- 웨이브 기반 밤 페이즈는 `타이머 종료`와 `남은 스폰 수`를 따로 두면 softlock가 나기 쉽다. 타이머가 끝난 뒤 남은 적이 사라지는지, 계속 스폰되는지, 0으로 정산되는지 종료 조건을 명시적으로 닫아야 한다.
- 계산대 상호작용은 `플레이어가 카운터 근처에 있는가`만 보지 말고, 맨 앞 손님이 실제 프런트 슬롯에 도착했는지까지 같이 확인해야 한다. 줄 상태와 결제 상태를 분리하지 않으면 아직 줄에 못 선 손님도 결제되는 버그가 생긴다.
- 줄 시스템은 스폰 id 순서가 아니라 `현재 줄 상태`와 `프런트까지의 실제 위치`를 기준으로 정렬해야 한다. 또 페이즈 타이머가 끝나도 손님이나 적 같은 진행 중 개체가 남아 있으면 `closing/cleanup` 상태를 먼저 거쳐 삭제나 누락 없이 종료시켜야 한다.
- 사용자가 `이전 요청은 잘못 썼다`고 방향을 정정하면, 진행 중이던 조사나 계획을 이어붙이지 말고 즉시 새 목표 기준으로 plan과 도구 선택을 리셋한다.
- 데스크톱에서 `한 화면 플레이`를 노리더라도 전역 `body` 스크롤을 기본으로 막아두면 안 된다. 한 화면 가정이 조금만 깨져도 핵심 UI가 접근 불가능해지므로, 실제 높이를 검증하기 전에는 `overflow-y: auto`를 유지하는 쪽이 안전하다.
- 카드 게임 UI에서 규칙 설명을 `설정 화면의 소개 문장`에만 두면 안 된다. 실제 플레이 화면에는 phase마다 `지금 할 일`, `왜`, `입력`을 한 묶음으로 다시 보여 주는 코치 패널이 필요하다.
- STS류 레퍼런스를 따라갈 때는 카드 테두리 색감만 닮게 만들지 말고, 먼저 `상단 얇은 HUD`, `중앙 전장 좌우 배치`, `하단 부채꼴 손패`, `보조 정보는 측면/별도 화면`이라는 공간 구성을 옮겨야 장르 감각이 난다.
- 사용자가 `아직도 길다`, `텍스트 위주다`라고 다시 지적하면 문장 다이어트만 하지 말고, 데스크톱 기준 `뷰포트 고정 높이`, `패널 내부 스크롤`, `설명문 -> 칩/수치`로 표현 형식 자체를 바꿔야 한다.
- 카드 배틀 HUD는 친절함을 이유로 코치 카드, 덱 설명, 인텔 설명, 로그를 모두 카드형 문단으로 쌓으면 바로 세로 길이가 터진다. 상단은 스트립, 측면은 요약 행, 로그는 최근 몇 개만 남기는 쪽이 기본값이어야 한다.
- 상단 HUD를 CSS grid로 짤 때 `span 2` 칩을 무심코 섞으면 데스크톱에서도 바로 두 줄이 생긴다. 핵심 전장을 한 화면에 유지해야 하는 게임에서는 HUD 칩 총 칸 수를 먼저 계산하고, 긴 값은 폭을 늘리지 말고 ellipsis로 처리하는 쪽이 안전하다.
- 사용자가 `진짜 게임처럼`을 요구하면 기존 대시보드 화면을 억지로 압축하지 말고, 먼저 `기본 씬`, `전투`, `선택 모달`로 상태를 분리해야 한다. 전투와 보상/휴식/대장간을 같은 캔버스에 동시에 두면 결국 웹 툴처럼 보인다.
- 추가로 사용자가 `사이드바로 돌아가지 말라`, `한 페이즈에 한 포커스`를 명시하면, HUD/로그/덱 정보는 선택 화면 안의 보조 칩 수준으로만 남기고 기본 레이아웃에서는 과감히 지워야 한다. 화면에 남는 것은 그 페이즈의 주 행동만이어야 한다.
